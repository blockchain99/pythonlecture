      <h4 class="section-heading">
        <a href="/blog/benefits-of-code-reviews">The Benefits of Code Reviews</a>
      </h4>
      <div class="card">
        <time pubdate="" datetime="2019-03-29"></time>
        <p>Code reviews can be a common practice for many developers: another engineer, often more experienced with the language, library, or code base being used, meets with the author-developer to read through their work, ask questions, and provide suggestions for improvement.</p>

<p>This process can be a positive experiment for many, but often, it can be challenging to be on the receiving end of a review. These can descend into demands for  needless uniformity of style (&quot;no! it&#39;s super-important to align your code <em>just so</em>&quot;) or they can create uncomfortable or harmful power dynamics between peers about whose style or ideas are &quot;better.&quot; At their very worst, code reviews can merely provide cover for biased viewpoints or microaggressions directed toward developers who may have less power in organizations.</p>

<p>While we recognize that not all parts of this practice are always helpful, we do encourage a lot of code reviews for our students here at Rithm. We think that, especially for newer developers in a class, these can provide useful feedback and lessons that can be difficult to get from lectures or books, and can also be hard to notice just from reading other code.</p>

      <h4 class="section-heading">
        <a href="/blog/three-more-js-mistakes-that-beginners-make">Three More JavaScript Mistakes That Beginners Make</a>
      </h4>
      <div class="card">
        <time pubdate="" datetime="2019-03-07"></time>
        <p>Despite its best intentions, JavaScript is not the most beginner friendly language. After working with thousands of beginners, here are a few common mistakes we see with applicants and students.</p>

<p>A little while back, we wrote a post on <a target="_blank" href="/blog/three-more-js-mistakes-that-beginners-make" rel="noopener noreferrer">the top four javascript mistakes beginners make</a>. In this post, we&#39;ll be adding a few more we&#39;ve seen to help you better prepare when learning the basics!</p>

<h3><strong>1. Dot vs Bracket notation</strong></h3>

<p>Let’s imagine we have the following object,</p>
<div class="CodeRay">
  <div class="code"><pre><span class="keyword">var</span> instructorData = {
  <span class="key">firstName</span>: <span class="string"><span class="delimiter">&quot;</span><span class="content">Elie</span><span class="delimiter">&quot;</span></span>,
  <span class="key">employer</span>: <span class="string"><span class="delimiter">&quot;</span><span class="content">Rithm School</span><span class="delimiter">&quot;</span></span>,
  <span class="key">favoriteNumber</span>: <span class="integer">42</span>,
  <span class="key">isHilarious</span>: <span class="predefined-constant">false</span>
};
</pre></div>
</div>

<p>When accessing a key in an object, there are two possible ways to go about it.</p>

<ul>
<li><p>dot notation</p></li>
<li><p>bracket notation</p></li>
</ul>

<p>Knowing when to use which one is essential, so here is a good rule of thumb to follow.If you know with 100% certainty what the key is - always use dot. If you are not 100% sure what the key will be, you <strong>must</strong> use bracket.</p>
